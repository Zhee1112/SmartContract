## High Severity

### 1. Reentrancy in UnoptimizedBridge.withdraw()

**Severity: HIGH** | **Detector: reentrancy-eth**

`UnoptimizedBridge.withdraw()` makes an external call (`msg.sender.call{value: amount}()`) BEFORE updating state (`balances[msg.sender] -= amount`). This allows recursive exploitation via `receive()`.

```
External call: msg.sender.call{value: amount}() (UnoptimizedBridge.sol#49)
State update after call: balances[msg.sender] -= amount (UnoptimizedBridge.sol#54)
```

**Status:** INTENTIONAL — This is the Tier [A] baseline contract demonstrating vulnerability. Exploited by `Attacker.attackUnoptimized()`.

## Medium Severity

### 2. Reentrancy in VictimBridge.withdraw() (Cross-function)

**Severity: MEDIUM** | **Detector: reentrancy-no-eth**

VictimBridge.withdraw() calls external monitor functions before updating `u.balance`. However, the EWS (MonitorMock) uses EIP-1153 TSTORE/TLOAD for callDepth tracking, which auto-resets per tx. The `nonReentrantTransient` guard in the withdraw path provides active protection.

```
External calls: monitor.recordTransaction(), monitor.enterCall(), monitor.checkAnomaly(), monitor.exitCall()
State update after calls: u.balance -= amount (VictimBridge.sol#126)
```

**Status:** MITIGATED by EIP-1153 transient storage reentrancy guard + CEI pattern.

### 3. Reentrancy in VictimBridge.swapETHForTokens() (Benign)

**Severity: LOW** | **Detector: reentrancy-benign**

State updates (`reserves.ethReserve`, `reserves.tokenReserve`) happen after external monitor calls. However, these are pool reserves, not user balances, and the monitor is a trusted contract.

**Status:** ACCEPTABLE — Monitor is a trusted contract deployed alongside the bridge.

### 4. Dangerous Strict Equality in MonitorMock.checkAnomaly()

**Severity: MEDIUM** | **Detector: incorrect-equality**

`lastTx.blockNumber == block.number` uses strict equality. In theory, a miner could manipulate block numbers, but in practice this is the core detection mechanism for sandwich attacks (comparing txs within the same block).

```
Condition: lastTx.blockNumber == block.number && lastTx.txType == 0
```

**Status:** INTENTIONAL — This is the sandwich detection logic. Same-block comparison is by design.

## Low Severity

### 5. State Variables Should Be Immutable

Several state variables set once in constructor should be `immutable`:

| Contract | Variable | Line |
|----------|----------|------|
| MonitorMock | `admin` | 32 |
| VictimBridge | `monitor` | 47 |
| UnoptimizedBridge | `admin` | 11 |
| Attacker | `unoptimizedBridge`, `victimBridge` | 20, 22 |

**Status:** Partially fixed — VictimBridge.admin is already immutable. MonitorMock.admin and VictimBridge.monitor should be made immutable.

### 6. State Variables Should Be Constant

| Contract | Variable | Line |
|----------|----------|------|
| BridgeWithSSTOREGuard | `_NOT_ENTERED`, `_ENTERED` | 19, 20 |

**Status:** LOW PRIORITY — These are SSTORE guard constants, cosmetic fix only.

### 7. Solidity Version Constraint Issues

`^0.8.20` contains known severe issues (VerbatimInvalidDeduplication, FullInlinerNonExpressionSplitArgumentEvaluationOrder, MissingSideEffectsOnSelectorAccess). MonitorMock uses `^0.8.24` (different from others).

**Status:** LOW RISK — These issues affect very specific edge cases not present in our contracts. All compiled with 0.8.28.

### 8. Low-Level Calls

All bridges use `msg.sender.call{value:}()` for ETH transfers. This is standard practice for reentrancy-safe patterns (CEI).

### 9. Assembly Usage

MonitorMock uses inline assembly for TSTORE/TLOAD opcodes (EIP-1153). This is required — no Solidity equivalent exists for transient storage.

### 10. Naming Conventions

`P_detect`, `_NOT_ENTERED`, `_ENTERED` don't follow mixedCase convention. These are domain-specific naming choices.
